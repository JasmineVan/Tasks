This list is used to manage object change history settings.

To enable object change history, in the Administration - Common settings - Change history section, select the Store change history check box (selected by default).

This will enable the Storage settings command, which is used to configure storing.

The list includes the following columns:

  • Object - list of applied objects displayed as a tree, grouped by object type.
  • When to save versions - selected option for saving versions;
  • Version lifetime - selected version lifetime.

Select object change history storage option

  • Select one or more objects, then click Set when to save versions, and select a preferred version storage option for each document and catalog type from dropdown list. You can also find the command in the More menu or right-click an object to open a context menu with this command.
    • Never - object version history is not maintained;
    • On write - a record is added to the version history whenever a document or catalog is created or modified;
    • On post (for documents) - a new version is saved on posting. Only applicable for documents that can be posted;
    • On start (for business processes) - a new version is saved when a business process starts. Only applicable for business processes;
    • Default - recommended settings for catalogs (never - do not generate versions), documents (generate versions on post), and business processes (generate versions on start).
  • Version storage settings can be modified for multiple applied objects of the same type at once. For example, you can change these settings for all documents or for all catalogs.

Select version lifetime

  • Click Set version lifetime and select the preferred version lifetime for each document and catalog type from dropdown list: You can store versions For the last week, For the last month, For the last three months, For the last six months, For the last year, and for an Unlimited period.

View the Number and size of stored object versions report

  • Use the Number and size of stored object versions link to view the Number and size of stored object versions report.

Clear obsolete versions

  • The Total obsolete versions field provides information on the number and size of obsolete versions in the application.
  • Click Clear to delete obsolete versions. The button is available only if there are obsolete versions in the application.
  • To Delete obsolete versions automatically, select the Delete obsolete versions automatically check box.
  • Follow the Set up schedule link to Set up schedule. The configured schedule is displayed at the bottom of the window.

Applying object version storage

  • During the early deployment stages, when data volume is relatively low and human errors (entering incorrect data, mistakenly deleting valid data, etc) are likely, storing versions can help you keep your documents and lists in order, rapidly eliminate the errors, and educate the application users.
  • Storing object versions allows you to easily roll any document or catalog item back to an earlier version that does not contain any errors or missing data.
  • As data volume tends to increase with time, it is feasible to limit storage of some object versions to the critical operations, such as document posting, or to waive the practice altogether.
  • Storing versions for a large number of applied objects requires significant disk space and deteriorates application performance. We recommend that you use this feature selectively, balancing advantages against performance issues on a case-by-case basis.

Notes on the On post method

  • If the method is set to On post for a document, no document versions will be saved until the document is posted for the first time. Once the document has been posted, new versions will be saved normally upon any change. This behavior is deliberate; it helps to avoid saving multiple versions of unposted documents, thus saving storage space. We recommend that you apply this setting to all documents.

Notes on the On start method

  • If the method is set to On start for a business process, no versions will be saved until the business process started for the first time. The method helps to avoid generating redundant business process versions while business processes are being prepared for the start.
