Use this form to describe user properties.

To enter or edit user details, you need administrator rights or user list manager rights.

Authorization of a user requires administrator's approval.

A user list manager can edit the user list (to a limited degree), configure users prior to their authorization approval by administrator, edit the Users list and the User groups list, and edit user access rights.

Other users are only allowed to view their personal data, change their Password (if not restricted by administrator), edit their contact information, and select application interface language.

In a distributed infobase, authorization is configured for each infobase separately.

Setting and changing passwords

  • Users can view their profiles in the application.
  • An administrator or user can set a password or change a password if necessary in case they have rights to do it.
  • by clicking Set password. If the password is already set, you can change it. To change a password, click Change password and enter the new password.

Recommendations on creating passwords

A strong password:

  • Is not human-readable.
  • Contains at least 8 characters.
  • Includes:
    • Uppercase Latin letters.
    • Lowercase Latin letters.
    • numbers
    • Special characters (such as underscore and parentheses).

Passwords that contain numbers only, human-readable words, or periodic character sequences, as well as passwords identical to usernames are not recommended.

  • Examples of strong passwords:
    “nj7{jhjibq*Gfhjkm
    F5”njnGhkmNj;t{HI
  • Examples of weak passwords:
    Smith
    qwerty
    12345678
    123123123

Creating users (for users with sufficient rights)

After a user list manager creates a user and specifies their username, a recommendation to approve their authorization settings is displayed to administrators in the user profile. Once an administrator approves authorization for this user, the recommendation is hidden and the manager is no longer able to change the authorization settings in any way, with the exception of revoking the authorization approval. To view the list of users that require approval, filter the user list by the Requires approval field.

The user details window contains the following tabs:

  • Main
    • Specify the Full name of the user. This name will be displayed in application lists and reports.
    • The Invalid check box is cleared for all users by default. If a user no longer uses the application (for instance, due to resignation) yet cannot be deleted due to being referenced by other objects, it is recommended that you select the Invalid check box for them. An invalid user is not displayed in the user list, and they cannot be selected, for example, as task performer or person responsible for a document. To restore the user to a valid status (for example, they have been rehired), clear this check box.
    • Username. A short name used to sign in to the application. It can differ from the full name (a user can have username "JSmith" and full name "John Smith").
    • 1C:Enterprise authentication. This is the default option.
      • 1C:Enterprise authentication status is displayed under the check box.
    • To set a password, click Set Password and enter it. Never leave this field blank as it compromises security.
      • If the user is not saved, you will be prompted to save it. Click Save.
      • Once the password is set, the authentication status changes to The password is set.
    • Require to change a password when signed in. Select this check box to require the user to set a new password when they sign in for the first time. This check box is selected for all users by default. To ensure data security, it is recommended that all users change the passwords preset by administrators when they sign in for the first time.
    • Show in selection list. Indicates whether the username is added to the selection list displayed at the application startup. If any external users are registered in the application, it is recommended that you clear this check box. If this check box is cleared, the user will be hidden from the list displayed at the application startup.
    • User cannot change password. Select this check box to prohibit the user from changing their password. In this case only administrator can change the password. This makes the Set password and Change password buttons unavailable to the user.
    • OpenID authentication. Select this check box to allow the user to sign in using an OpenID account. This feature is used in 1C:Enterprise web client. If you select this check box, also select the 1C:Enterprise authentication check box. Otherwise the application might work incorrectly.
    • OS authentication. Select this check box for quick and easy authorization.
      • In the User field, click and select the operating system username. The user will be able to sign in to the application with this username. If you select this check box, you can clear the 1C:Enterprise authentication check box.
    • If none of the Authentication check boxes are selected, the Can sign in check box is automatically cleared.
    • Run mode. The default value is Auto. This is the recommended mode.
  • Addresses and phones
    • Use this tab to store contact information of the user (for instance, phone number and email address).
  • Comment
    • Use this tab to store additional information about the user.

Adding users to groups

If the application supports user groups, you can add users to one or several groups during the user creation procedure.

  • Click Groups on the navigation panel, and select the check boxes for one or several groups.
  • To clear all selections quickly, click Remove from all groups.

Access rights of users

Use one of the following options, provided that it is available in the application:

  • Click Access rights on the navigation panel to add a user to one or more access groups. Select the groups from the list.
  • In the Allowed actions (roles) list, assign one or several roles to the user, thus defining their personal access rights settings. The roles can be linked to job duties of users or to smaller activities.
    • Select the check boxes for the roles you want to assign.
    • To view selected roles only, click Show selected roles only.
    • You can sort roles alphabetically or group them by subsystems.

Allowing authorization (for administrators only)

  • Select the Authorization allowed checkbox to allow a user to sign in to the application Then set the username, password, and other authentication parameters.
  • You can deny user access to the application manually by clearing the Allow authorization check box, or have the access denied automatically at the end of the specified period.
  • Open user sessions are closed automatically no longer than 25 minutes after the user access is denied.

Authorization restrictions

Follow the link to edit authentication restriction. Select one of the following:

  • Apply general authorization settings. This is the default option.
  • No time limit.
  • Allow authorization till. Enter the date or click to select it from the calendar. To prevent unauthorized access to the application, you can set an Expiration date. Upon reaching this date, the user is denied access to the application.
  • Deny authorization if no activity for (specify the number of days). If a user does not sign in for more than the specified number of days, they are denied access to the application.
    • In this case the user has to contact the administrator to restore access to the application.
  • Restrictions are listed in user profiles. Restrictions for all users are also listed in the User details report.

Managing user settings

  • The application stores individual settings of each user for further use. To view, copy, or clear user settings, click Settings on the navigation panel. Clearing might be required when invalid settings affect normal operation of the application.

Application warnings

  • If application user properties differ from the properties of the matching infobase user, the application displays a warning including a list of differences. Then it prompts you to save the data in order to resolve the differences.
