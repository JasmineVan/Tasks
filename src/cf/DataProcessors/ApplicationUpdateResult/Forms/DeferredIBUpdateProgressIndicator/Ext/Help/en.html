This form is used to display progress and view results of the deferred infobase updates.

When performing infobase version updates, you may choose to have some of the data updated in deferred mode, i.e. in background, without service interruption. In this mode, the documents that have not been processed yet will become temporarily unavailable. If you try to open any of such documents, the program will display the message warning that actions with documents are blocked for the time the application is being updated. Some reports may also become unavailable during the update. When a user attempts to run a report, they get a warning that the report might contain inaccurate data because the update is in progress.

The form and the () window open simultaneously and automatically the moment the user signs in to the application after the infobase update.

Setting priorities

  • The application allows to make data processing more quick by setting higher priority for it then the user work. This accelerates the update.
  • The check box allows to set priorities for data processing:
    • —data is being processed slowly, diminishing the speed of the server ans users work less.
    • —data is being processed quickly, diminishing the speed of the server ans users work.
      • If priority is given to data processing, there is no delay between executions of scheduled data processing jobs.
      • Use this option when access to data that is being processed has to be retrieved quickly.
  • Giving high priority to additional data processing automatically gives the same priority to the procedures it depends on.

Multithreaded update

  • Update can be made in multiple threads, loading all central processing unit cores by switching the radio button to .
  • During the multithread update, the scheduled job controls the update process (starts background jobs). Only this scheduled job addresses common resources. Only this scheduled job addresses common resources. Separate background jobs are started for each part of the update process: no more than N jobs at once, where N is the number of central processing unit cores. Background jobs set environment, execute an update handler, and save results.
  • Define the total number of central processing unit cores on all working servers of a cluster: for instance, if there are 2 working servers, each possessing 8 cores, the total number is 16.
  • Give high priority to data processing for the update;
  • Define the number of threads equal to the number of central processing unit cores;
  • Set the update schedule (for instance, to non-working hours, not to interrupt user sessions).
    • Number of threads and schedule can be changed anytime. These changes are applied to the update process immediately:
    • If the number of threads gets higher, more background tasks are started;
    • If the number of threads gets lower, redundant background tasks are finished and new background tasks are not started;
    • You can set the schedule so that update will be in progress only at certain hours: background and scheduled jobs will be finished before a set hour.

Setting data processing schedules

  • Data processors often take a long time to complete (from several minutes to hours, depending on volume of data stored in the application and computer performance). Moreover, while data processors are running the user experience may suffer. It is recommended to schedule data processing for hours of minimum user activity.
