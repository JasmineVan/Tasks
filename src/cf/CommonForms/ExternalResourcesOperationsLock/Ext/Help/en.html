When copying files of the infobase to another location, both the original and copied infobases can continue to interact with external resources. For example, report mailing, data synchronization with other applications, and sending or receiving email. To prevent it, when changing the location of the infobase, all such scheduled jobs are locked. When signing in to the moved infobase first time, administrator needs to make a decision on infobase role: If infobase is moved, operations with external resources is resumed. If it is infobase copy, operations with external resources is locked.

Determining whether infobase is moved or not

When determining whether infobase is moved or not, first of all, change in type of infobase location is checked. From the file type to client/server type and vice versa. If the location type is changed, it is considered moved. If the location type is not changed, the infobase connection string is compared. If the connection string is not changed, the further check is stopped and the infobase is considered not moved. Further check depends on the type of infobase location.

  • In the file infobase, you can determine whether infobase is moved or not by the DoNotCopy.txt service file, automatically created by the application in the infobase directory. There is infobase ID in the file. If there is no ID or ID in the file is different from ID in the infobase, the infobase is considered moved.
  • In the client/server infobase, you can determine whether infobase is moved or not:
    • If infobase name was changed. For example, the infobase name was "base_bp" and became "base_bp_test".
    • You can also determine it, if the computer or server name was changed. For example, the computer name was "main_server" and became "test_server", and the server name was "main" and became "test".

    When operating in SaaS, Service manager sets the flag that shows whether the application is moved or not. If the infobase is imported in the application or exported from it, it is determined automatically.
